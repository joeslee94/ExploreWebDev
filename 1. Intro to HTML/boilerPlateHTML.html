    Background information
  • *This article if from '03'*
  • UTF-8 is staying and not going anywhere
  • Creating software that handles international input must be hard
  • ASCII is 32 to 127 with a space being 32 "A" being 65 and "a" being 97
  • This system worked so long as we spoke English
  • 128-255 were free real estate, so IBM, people from other countries, and anyone else took advantage
  • So code pages were created to handle these differences BUT still limitations
  • *Couldn't use s++ or s-- to iterate through a string, but rather Windows.AnsiNext and Prev*
    Unicode
  • Every platonic letter in every alphabet is assigned a magic number
  • Magic number example: U+0639 and is known as a code point
  • "Hello" == U+0048 U+0065 U+006C U+006C U+006F
  • But how is this stored in memory? Or how is it supposed to be represented in an email?
    Encodings
  • 00 48 00 65 00 6C 00 6C 00 6F == 48 00 65 00 6C 00 6C 00 6F 00
  • FE FF notations since there were two ways of storing Unicode
  • That's where UTF-8 came in
  • Bunch of ways to encode Unicode (some more outdated than others): UCS-2, UTF-16, FE/FF UCS-2, UCS-4, UTF-7 UTF-8
    Closing Remarks
  • Makes no sense to have a string withou tknowing what encoding it uses
  • There ain't no such thing as plain text
  • As an example, must have this in the email to make it read correctly: Content-Type: text/plain; charset="UTF-8"
  • Overall need to make sure we are accounting for the encoding of the text we are receiving and sending
